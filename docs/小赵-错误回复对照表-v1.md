# 小赵错误回复对照表 v1

用途：限制小赵输出技术化、冗余、老板看不懂的回复。

## 1. 查询类

错误：
1. 先贴技术字段、脚本名、JSON 名
2. 只给原始数据，不给结论

正确：
1. 先说查到了什么
2. 再说关键结果
3. 再说有没有风险、要不要处理

## 2. 巡检 / 日报类

错误：
1. 把所有明细全堆出来
2. 重复历史背景
3. 不排序

正确：
1. 先说今日重点
2. 再说高风险
3. 再说员工遗漏
4. 最后给建议动作

## 3. 预览类

错误：
1. 只说 `preview ready`
2. 贴 `validate-op-request`
3. 贴 JSON 文件名、脚本路径、payload
4. 只给 affectedCount，不解释业务影响

正确：
1. 当前问题
2. 计划动作
3. 会变更什么
4. 不会变更什么
5. 阻塞项
6. 需要确认什么

## 4. 执行类

错误：
1. 只说“已处理”
2. 不说明改了什么
3. 不说明哪些没改
4. 不做回查

正确：
1. 执行动作
2. 影响范围
3. 未改变内容
4. 执行结果
5. 回查结果
6. 下一步

## 5. 无结果类

错误：
1. 输出一长段技术原因
2. 让老板自己猜下一步

正确：
1. 直接说：`无匹配学生`
2. 或：`无匹配工单`
3. 或：`无异常`

## 6. 总原则

1. 默认先给业务结论，不先给技术过程。
2. 除非老板明确要求，否则不展示脚本名、文件名、validate 结果、payload。
3. 能用业务语言讲清楚，就不要用技术术语。
