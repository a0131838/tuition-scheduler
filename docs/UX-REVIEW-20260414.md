# UX Review 2026-04-14

## Goal

Review the mature internal system from real operating flows, not just visual polish, and improve it through gradual changes that reduce friction without rewriting the product.

## Key Flows Reviewed

- admin dashboard
- approval inbox
- expense claims workflow
- receipt queue and package finance workspace
- student list
- student detail
- scheduling coordination workspace

## Overall Diagnosis

- The system is functionally rich and workflow-aware.
- The biggest problems are no longer missing features; they are scanning cost, repeated chrome, heavy first screens, and inconsistent local interaction patterns.
- High-frequency users lose time because they must read too much before acting, confirm the current data scope manually, and re-learn the switching pattern on different pages.

## Highest-Impact Problems

1. Shared page chrome is too heavy
- many pages repeat workspace explanation, top controls, summary cards, and secondary navigation before showing the real work area
- first-screen efficiency suffers on dashboard, approval inbox, receipt queue, and student work areas

2. Queue pages are still too presentation-heavy
- approval inbox and receipt queue rely on large cards and long descriptions
- this slows scanning, comparing, and clearing multiple items in one session

3. Current dataset scope is easy to misread
- student list can reopen a remembered desk and remembered filters
- users can mistake a narrowed result set for the full system dataset

4. Workflow isolation is incomplete
- the dedicated student coordination page still carries too much student-detail weight
- the system says “workspace,” but the page still feels like a long profile with a workspace appended

5. Status language is not yet action-first everywhere
- some views still make the user infer who acts next
- states, risk tags, and readiness tags are not fully separated conceptually

## Execution Plan

### Batch 1: done first

- compress dashboard first-screen messaging and prioritize today’s real work
- make approval inbox more scan-friendly and more list-like
- make student-list dataset scope and restored-filter state explicit

### Batch 2: next

- slim down receipt queue cards into a denser processing list
- move expense-approval configuration out of the main first-screen action area
- add stronger “back to source workflow” paths from inbox and queue driven pages

### Batch 3: later

- convert student coordination into a true light-shell workspace
- unify status badge language across approval, receipt, and queue flows
- unify secondary navigation patterns across workbench pages

## Page-by-Page Recommendations

### Dashboard

- keep only the most important work starters in the first screen
- move lower-frequency setup and reference entry points lower
- use compact operational metrics instead of large explanatory sections

### Approval Inbox

- use compact rows instead of large cards
- show filter counts before users switch focus
- keep current scope visible
- prioritize amount, lane, urgency, and risk over long descriptions

### Expense Claims

- separate approval work from approval configuration
- keep the first screen focused on submitted items, overdue items, and attachment issues

### Receipt Queue

- reduce queue-card text and increase comparable fields
- keep detailed proof, invoice, and timeline content in the detail pane only

### Student List

- strongly signal when the system restored the last desk
- always show whether the page is narrow scope or full list
- give one-click recovery to the full list

### Student Coordination

- keep only student identity essentials in the workspace shell
- move full profile context behind an expandable snapshot

## Release Strategy

- ship gradual UX improvements in small batches
- prefer layout, wording, and interaction refinements over structural rewrites
- avoid mixing UX cleanup with business-rule changes in the same release
