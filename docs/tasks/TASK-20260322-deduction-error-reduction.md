## 主要发现（高优先级）

### 1) 两条保存链路存在逻辑漂移（高风险）
- `attendance/route.ts` 与 `mark-all-present/route.ts` 各自维护一套扣减判断与流水一致性检查。
- 已观察到 `EXCUSED + excusedCharge` 相关判断在两处并不完全一致。
- 影响：同一节课在“逐个保存”和“一键全到”下，可能出现扣减结果不一致。

### 2) 前端告警是“提示型”，但允许继续提交（中高风险）
- AttendanceEditor 中存在“余额不足/异常值”的提示，但当前并非强约束拦截。
- 影响：高频操作时，教务可能在告警条件下仍提交，造成后续人工修正。

### 3) 状态切换触发自动回填，容易被误改（中风险）
- 学生状态、课包、扣减单位切换时有自动回填逻辑（分钟/次数）。
- 影响：快速操作中，人工输入值可能被覆盖或与预期不一致。

## 低风险改进建议（不改变核心业务规则）

### 建议 A：统一扣减判定函数（优先）
- 把两条 API 链路的判定逻辑抽到同一共享函数（只做“抽取复用”，不改判定规则）。
- 目标：保证“逐个保存”和“一键全到”输出一致，避免双维护漂移。

### 建议 B：把关键告警升级为“可配置拦截”
- 默认保持现状（只提示），新增可开关“严格模式”：
  - 严格模式 ON：关键异常（如余额不足、非法负值）禁止提交。
  - 严格模式 OFF：维持当前行为。
- 目标：先不改变既有学校习惯，给管理员可控开关。

### 建议 C：提交前增加“扣减摘要确认”
- 在保存前显示简短摘要：本次将扣减 X 人、总分钟/总次数 Y、异常 Z 人。
- 目标：降低误操作，尤其是“一键全到”。

### 建议 D：自动修复保留，增加“已自动调整”标记
- 不改自动修复逻辑，仅在 UI 显示“本行被系统调整过”标记与原因。
- 目标：让教务知道值为何变化，减少“明明我填过”的困惑。

## 影响边界说明（给审批）
- A 为代码结构优化，理论上不改变业务行为；主要风险是重构引入回归，需要回归测试。
- B/C/D 为交互层增强，不改变账务扣减规则；风险集中在前端流程感知变化。
