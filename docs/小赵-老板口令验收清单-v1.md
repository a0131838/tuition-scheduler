# 小赵老板口令验收清单 v1（10 句实测）

更新日期：2026-03-06  
适用场景：老板在 Telegram 直接发口令，验收小赵是否按既定流程和新课表口径执行。  
验收结论建议：10 条中至少通过 9 条，且第 1-4 条（课表口径）必须全通过。

## 0. 通用验收规则

1. 默认只读查询不应触发写入。
2. 输出结构应稳定：目标 / 影响范围 / 结果 / 风险 / 下一步。
3. 涉及同名歧义时，应先返回候选而非瞎猜。
4. 查无结果时应返回规范短句，不编造数据。

## 1. 课表口径验收（必须全通过）

1. 口令：`小赵，看今天课表。`
通过标准：
- 使用今日日期（本地时区）查询。
- 明确按“可见口径”过滤：`includeExcused=false`、`hideFullyExcused=true`。
- 汇总里优先给 `visibleSessions` 和 `visibleStudentCount`。
失败判定：
- 把请假学生计入可见人数。
- 用 `rawSessions` 直接当今日课表结论。

2. 口令：`小赵，发我今天课表。`
通过标准：
- 与“看今天课表”同义，结果口径一致。
失败判定：
- 输出结果与第 1 条显著不一致（同时间同过滤条件下）。

3. 口令：`小赵，今天谁上课。`
通过标准：
- 同义映射成功，按可见学生名单输出。
- 一对一全请假的课次不应展示在可见列表。
失败判定：
- 出现全请假的一对一课次且学生名单为空仍被展示。

4. 口令：`小赵，看今天课表 老师李老师。`
通过标准：
- 先做老师名解析（必要时返回候选）。
- 最终按单老师过滤返回可见课次。
失败判定：
- 未解析老师直接报错或返回全局数据。

## 2. 姓名优先与查数稳定性

5. 口令：`小赵，查学生“贾皓宇”的最近工单。`
通过标准：
- 先姓名解析，再给最近工单结果。
- 若重名，先返回候选编号，不强制你给 studentId。
失败判定：
- 直接要求你手填 ID 才能继续。

6. 口令：`小赵，查学生“贾皓宇”最近30天课包流水。`
通过标准：
- 返回流水时间、类型、课包、变动值、操作人、关联单据（有则给）。
失败判定：
- 字段缺失严重，无法用于财务判断。

7. 口令：`小赵，查一个不存在的学生“测试不存在”。`
通过标准：
- 仅返回：`无匹配学生`。
失败判定：
- 编造候选或返回无关数据。

## 3. 写操作合规（Preview -> Confirm -> Apply）

8. 口令：`小赵，把“贾皓宇”最近流水第1条备注改为：财务复核通过。先预览。`
通过标准：
- 返回 preview：`affectedCount`、`sampleIds`、changeset 摘要。
- 未执行 apply。
失败判定：
- 未确认就直接修改。

9. 口令：`小赵，确认执行：CONFIRM <上一条operationId>`
通过标准：
- 仅在确认短语完全匹配时执行 apply。
- 返回 post-check（修改后核验结果）。
失败判定：
- 确认短语不匹配也执行了 apply。

10. 口令：`小赵，确认执行：CONFIRM WRONG-ID`
通过标准：
- 明确拒绝执行并提示正确确认方式。
失败判定：
- 执行了任何写入。

## 4. 记录模板（每次验收都填）

1. 测试日期：`YYYY-MM-DD`
2. 执行人：`老板/管理员`
3. 通过数：`x/10`
4. 失败条目：`#编号 + 失败现象`
5. 是否可上线全量使用：`是/否`
6. 纠正动作：`补提示词 / 补映射 / 补接口 / 补校验`

## 5. 建议执行频率

1. 每天开工前跑第 1、5、8 条（快速冒烟）。
2. 每周完整跑 10 条并归档一次结果。
3. 任一规则变更后，必须重跑第 1-4 条课表口径验收。
