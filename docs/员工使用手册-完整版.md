- `周课表`：`/admin/schedule`
- `冲突中心`：`/admin/conflicts`

## 1. 业务对象关系（先理解再操作）

- 校区 `Campus` -> 教室 `Room`
- 课程体系 `Course/Subject/Level`
- 老师 `Teacher`（授课能力 + 可上课时间）
- 学生 `Student`
- 课包 `Package`（可排课、可扣费的前提）
- 班级 `Class`
- 课次 `Session`
- 报名 `Enrollment`
- 点名 `Attendance`
- 反馈 `Session Feedback`
- 预约/链接 `Booking Link/Request`
- 工单 `Ticket`

关键原则：
1. 没有有效课包，不能正常排课/报名。
2. 排课必须同时满足：老师可用时间、老师无冲突、教室无冲突、预约无冲突。
3. 点名与扣费流水必须可追溯。

## 3. 教务日常 SOP（Admin）

### 3.1 开工顺序（固定）
1. 打开 `/admin/todos` 清 `昨日/今日未点名`。
2. 处理 `/admin/conflicts`，先把冲突降到可控。
3. 进入 `/admin/schedule` 做当日排课与改课。
4. 必要时用 `/admin/schedule/new` 创建单次课/预约。

### 3.2 排课动作清单
- 班课：`/admin/classes` -> `/admin/classes/[id]/sessions`
- 一对一：`/admin/students/[id]` 快速排课
- 换老师：学生页/班级课次页/课次页入口
- 取消与恢复：学生页课次记录或课次点名页

### 3.3 冲突处理标准
在 `/admin/conflicts` 统一闭环：
- 老师冲突（课次/预约重叠）
- 教室冲突（同时间重复占用）
- 处理策略：换老师 -> 换教室 -> 调整时间 -> 取消/改约

## 4. 学生与课包 SOP（学管）

### 4.1 学生档案
- 列表：`/admin/students`
- 详情：`/admin/students/[id]`
- 重点检查：来源、类型、最近出勤、未来课次、可排课条件

### 4.2 课包管理
- 列表：`/admin/packages`
- 账本：`/admin/packages/[id]/ledger`
- 票据页：`/admin/packages/[id]/billing`

硬规则：
1. 同学生同课程同模式的有效期避免重叠。
2. 课包状态、余额、有效期必须和排课需求对齐。
3. 扣费异常先核对点名，再核对 ledger，再核对包有效性。

### 4.3 每日动作
1. `/admin/todos` 看续费预警。
2. 对预警学生补包/续费。
3. 抽样核对 ledger 与点名一致性。

## 5. 班级与报名 SOP（教务）

- 班级：`/admin/classes`、`/admin/classes/[id]`、`/admin/classes/[id]/sessions`

2. 报名学生（确保学生对应该课程有有效课包）。
3. 生成或新增课次。
4. 监控冲突并闭环。

### 7.3 财务闭环检查
1. 票据审批状态与收款状态一致。
2. 课包扣减与点名记录一致。
3. 结算单导出前确认审批链完整。
4. 工资报表与课时来源一致。

## 12. 常见报错与处理顺序

1. `No active package for this course`
- 先建/续课包，再排课。

2. `No eligible teachers`
- 查老师授课能力 -> 查可用时间 -> 查冲突。

3. `Outside availability`
- 调整老师可用时间或改排课时段。

4. `Teacher/Room conflict`
- 到 `/admin/conflicts` 统一处理，不在单页硬改。

5. 老师端看不到课次
- 核对老师账号是否绑定到对应 teacher 档案。

## 13. 小赵（OpenClaw）代操作规范

你要求“小赵可代执行 Admin + 财务”，必须使用以下规则：

1. 所有数据变更必须两阶段：
- 阶段 A：Preview（显示命中记录、影响条数、关键字段前后对比）
- 阶段 B：Apply（你明确确认后执行）

2. 小赵每次执行必须回报：
- 变更了什么
- 没变更什么
- 影响行数
- 复核证据（查询/日志摘要）

3. 高风险操作（删除、批量改财务数据、批量改排课）必须附回滚方案。

4. 小赵优先使用现有业务 API 与服务逻辑，不允许绕过业务规则直接破坏性改库。

5. 生产操作顺序：
- 先测环境验证
- 再生产 preview
- 你确认后 apply

6. 强制校验器（执行前必须通过）：
- 模板文件：`ops/codex/op-request.template.json`
- Preview 校验：
```bash
node ops/codex/validate-op-request.mjs --file <request.json> --mode preview
```
- Apply 校验：
```bash
node ops/codex/validate-op-request.mjs --file <request.json> --mode apply
```
- Apply 必须同时满足：`forceApply=true`、确认短语一致、`affectedCount <= maxAffected`。
